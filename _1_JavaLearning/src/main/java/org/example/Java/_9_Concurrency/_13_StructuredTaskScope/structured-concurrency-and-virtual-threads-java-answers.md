# Ответы: Structured Concurrency и Virtual Threads

Ниже ответы на вопросы из [structured-concurrency-and-virtual-threads-java.md](C:\Users\Admin\IdeaProjects\Learning\_1_JavaLearning\src\main\java\org\example\Java\_9_Concurrency\_13_StructuredTaskScope\structured-concurrency-and-virtual-threads-java.md).

Ответы ниже ориентированы в первую очередь на JDK 25, потому что в проекте модуль настроен на Java 25 preview. Там, где вопрос привязан к старому preview API, это отмечено отдельно.

## 1. Чем Structured Concurrency отличается от обычного Executor?

`Executor` просто запускает задачи, но не связывает их жизненный цикл с родительской операцией. `StructuredTaskScope` делает группу подзадач одной логической единицей работы: подзадачи открываются внутри блока, завершаются до выхода из него, а ошибки и отмена обрабатываются согласованно.

Практический эффект:

- проще отменять связанные подзадачи;
- легче корректно дождаться завершения всех дочерних задач;
- проще наблюдать причинно-следственную связь "родительская задача -> дочерние задачи";
- меньше риск утечек фоновых задач.

## 2. Что делает `StructuredTaskScope.ShutdownOnFailure`?

Это ответ для JDK 21-24 preview API. `ShutdownOnFailure` захватывает первую ошибку подзадачи, затем завершает scope, прерывает незавершенные подзадачи и будит поток-владельца scope.

В JDK 25 API изменился: типичного отдельного класса `ShutdownOnFailure` больше нет в основном сценарии, а `StructuredTaskScope.open()` по умолчанию открывает scope с политикой "ждать все успешные задачи или упасть на первой ошибке".

## 3. Как Structured Concurrency помогает управлять группой потоков и исключениями?

Она вводит структурированную область жизни задач:

- подзадачи fork-ятся только внутри scope;
- `join()` ждет завершения группы как единого целого;
- ошибка одной подзадачи не теряется, а становится частью результата `join()`;
- закрытие scope гарантирует, что незавершенные подзадачи не останутся "болтаться" после выхода из блока.

## 4. Как Structured Concurrency помогает избежать "зависших" потоков?

Потому что дочерние задачи существуют только в пределах scope. Родитель не может "случайно забыть" про них и выйти дальше по коду, как это часто бывает с обычным `ExecutorService` и `Future`, если кто-то не сделал `get()` или не отменил задачу.

## 5. Можно ли использовать виртуальные потоки внутри `StructuredTaskScope`? Если да, зачем это удобно?

Да. В JDK 25 это поведение даже используется по умолчанию: если не задан свой `ThreadFactory`, subtasks запускаются в unnamed virtual threads.

Это удобно, потому что:

- код остается линейным и блокирующим по стилю;
- можно делать много I/O-bound подзадач;
- structured concurrency дает контроль над группой задач, а virtual threads дают масштабируемость.

## 6. Сколько OS-потоков реально используется для 1000 виртуальных потоков?

Обычно не 1000. Виртуальные потоки исполняются поверх меньшего числа platform threads, которые выступают как `carrier threads`.

По документации JDK scheduler виртуальных потоков использует target parallelism, который по умолчанию равен числу доступных процессоров. Поэтому для 1000 виртуальных потоков число одновременно нужных carrier threads обычно ближе к `Runtime.availableProcessors()`, а не к 1000.

Это вывод по официальному описанию scheduler, а точное число в конкретный момент зависит от нагрузки, блокировок, pinning и настроек scheduler.

## 7. Что такое `carrier thread` и зачем он нужен?

`Carrier thread` это обычный platform thread, на который JVM временно "монтирует" виртуальный поток для выполнения.

Идея такая:

- виртуальный поток сам не равен OS-потоку;
- JVM планирует виртуальные потоки на carriers;
- когда виртуальный поток блокируется в поддерживаемой операции, он может "размонтироваться", а carrier освобождается для другой задачи.

## 8. Что происходит, когда виртуальный поток вызывает `Thread.sleep()`?

Он временно перестает исполняться. Для виртуальных потоков блокирующие операции JDK обычно позволяют освободить carrier thread, чтобы тот выполнял другие виртуальные потоки.

Для `Thread.sleep()` официальная документация `Thread` говорит только о временном прекращении выполнения, а документы по virtual threads описывают общий механизм mount/unmount для blocking operations в JDK. Поэтому здесь разумный вывод такой: сон виртуального потока не должен требовать выделенный OS-thread на весь период сна, если поток не pinned.

Это именно вывод из общего механизма virtual threads, а не отдельная формулировка только про `sleep()`.

## 9. Как JVM решает, когда возобновить виртуальный поток?

Когда завершается причина блокировки:

- пришли данные из I/O;
- истек timeout;
- произошло `unpark`/сигнал пробуждения;
- завершилась иная ожидаемая операция.

После этого операция повторно отправляет виртуальный поток в scheduler JVM, а тот монтирует его на carrier thread и продолжает выполнение.

## 10. Сколько обычных потоков понадобилось бы для 1000 клиентов с blocking I/O?

В классической модели "один запрос = один platform thread" потребовалось бы примерно до 1000 обычных потоков, если все 1000 клиентов действительно обслуживаются одновременно и каждый блокируется на I/O.

На практике это быстро упирается в память, стек каждого потока, scheduler overhead и контекстные переключения.

## 11. Почему использование виртуальных потоков эффективнее?

Потому что они намного дешевле platform threads и лучше подходят для задач, которые большую часть времени ждут I/O.

Преимущества:

- меньше накладные расходы на создание и хранение потоков;
- не нужно держать отдельный OS-thread на каждую ожидающую задачу;
- можно писать обычный блокирующий код без сложных callback/reactive-цепочек;
- выше масштабируемость на большом количестве одновременных I/O-bound задач.

## 12. Почему код выглядит линейным, но не блокирует OS-потоки?

Потому что блокируется именно виртуальный поток как логическая единица выполнения, а не обязательно platform thread под ним. Если операция поддерживает unmount, JVM снимает виртуальный поток с carrier thread и использует carrier для других задач.

То есть стиль кода синхронный, а исполнение под капотом M:N.

## 13. Что будет, если увеличить количество клиентов до 100000?

Для virtual threads это намного реалистичнее, чем для platform threads, но это не "бесконечная" масштабируемость.

Упрется в:

- доступную память;
- реальную пропускную способность БД, сети и внешних API;
- число открытых соединений;
- pinning и синхронизацию;
- лимиты scheduler и внешних ресурсов.

То есть virtual threads убирают узкое место "слишком дорогие потоки", но не убирают лимиты внешних систем.

## 14. Какие ограничения накладывает количество `carrier threads`?

Оно ограничивает, сколько виртуальных потоков могут реально исполняться на CPU одновременно. Если виртуальные потоки часто pinned или долго занимают carriers, пропускная способность падает.

Практически это значит:

- мало carriers -> больше очередь runnable virtual threads;
- частый pinning -> carriers хуже переиспользуются;
- CPU-bound задачи не получают магического ускорения от virtual threads.

## 15. Что такое `Scoped Values` и зачем они нужны?

`ScopedValue` это механизм для безопасной и эффективной передачи значения от вызывающего кода к его прямым и косвенным вызовам внутри потока, а также к дочерним потокам, без явной протяжки параметров и без `ThreadLocal`.

Они нужны, чтобы хранить контекст выполнения, например:

- user/session/request context;
- trace id;
- security context;
- локальные настройки обработки.

## 16. Чем `Scoped Values` отличаются от `ThreadLocal`?

Главные отличия:

- `ScopedValue` живет в ограниченной динамической области выполнения, а не "пока кто-то не очистит";
- binding у `ScopedValue` логически одноразовый и проще для рассуждения;
- `ScopedValue` лучше подходит для наследования контекста в structured concurrency;
- `ThreadLocal` проще случайно утечь или забыть очистить;
- `ScopedValue` дешевле по space/time cost, особенно вместе с virtual threads.

## 17. Почему `Scoped Values` особенно полезны для виртуальных потоков?

Потому что virtual threads могут быть очень многочисленными. `ThreadLocal` на больших количествах потоков становится дороже и сложнее для контроля. `ScopedValue` дает более легкий и структурированный способ передавать контекст.

Кроме того, `StructuredTaskScope` захватывает текущие scoped value bindings при создании scope и передает их fork-нутым задачам.

## 18. Как передать значение через несколько виртуальных потоков без явного `ThreadLocal`?

Через `ScopedValue.where(...).run(...)` или `call(...)`, а затем внутри этого блока открыть `StructuredTaskScope` и `fork()` подзадачи. Тогда дочерние потоки унаследуют binding автоматически.

Именно это и является рекомендуемым структурированным способом передачи контекста.

## 19. Что произойдет, если несколько виртуальных потоков изменяют одно и то же `Scoped Value`?

Сам `ScopedValue` не предназначен для произвольной общей изменяемой записи как `ThreadLocal map`. Binding scoped value делается на ограниченный участок выполнения и является per-thread.

Корректнее говорить так:

- разные потоки могут иметь разные bindings одного и того же `ScopedValue`;
- внутри вложенного scope можно сделать rebinding;
- после выхода из вложенного scope значение откатится к предыдущему binding.

Если же несколько потоков получили один и тот же изменяемый объект как значение `ScopedValue`, то гонки возможны уже на уровне этого объекта. Документация прямо требует либо immutable value, либо корректную синхронизацию доступа.
