# Angular Learning Plan: расширенный программный трек (много независимых страниц)

Дата актуализации плана: 27 марта 2026.

Этот документ специально построен как "каталог страниц-лабораторок":
- Каждая страница независима по бизнес-сценарию.
- Сложность растет по Angular-инструментам.
- Каждая страница заканчивается конкретным результатом: "страница с логикой".

## Актуальные ориентиры перед стартом

- Базовый ориентир: Angular v21 (актуальная ветка на момент проверки).
- Основной стиль разработки: standalone-компоненты и функциональные API.
- Для роутинга и тестов используем современный стиль через `provideRouter`.
- Для HTTP в конфиге приложения используем `provideHttpClient(...)`.
- Для интерсепторов в приоритете функциональные interceptors.
- Для тестов в новых CLI-проектах по умолчанию используется Vitest.
- Signal Forms в документации отмечены как experimental, поэтому в плане это отдельный optional-блок.

## Формат каждой учебной страницы

Для каждого маршрута делай одинаковый каркас:
1. `route` страницы.
2. Компонент страницы и шаблон.
3. Минимальная логика в TypeScript.
4. Проверка happy-path и одной ошибки.
5. Короткий итог: что изучил, что реализовал.

## Блок A. Базовые шаблоны и компоненты (страницы 1-9)

### 1) Компонент-основа (`/lab/component-basics`)
- Цель: понять, как создается standalone-компонент.
- Что кодим: заголовок, подпись, кнопка "обновить текст".
- Ключевые API: `@Component`, `imports`, шаблон, поля класса.
- Проверка: кнопка меняет текст в шаблоне.
- Итог: страница с логикой базового компонента.

### 2) Интерполяция и property binding (`/lab/binding-basics`)
- Цель: закрепить `{{ }}` и `[property]`.
- Что кодим: поле "name", отображение имени, блокировка кнопки по условию.
- Ключевые API: интерполяция, `[disabled]`, `[title]`.
- Проверка: кнопка активируется только при валидном состоянии.
- Итог: страница с логикой отображения состояния.

### 3) Event binding (`/lab/events`)
- Цель: научиться обрабатывать пользовательские события.
- Что кодим: счетчик с `increment`, `decrement`, `reset`.
- Ключевые API: `(click)`, методы компонента.
- Проверка: корректные переходы между состояниями счетчика.
- Итог: страница с логикой событий.

### 4) Control flow: `@if`, `@for`, `@switch` (`/lab/control-flow`)
- Цель: освоить новый синтаксис control flow.
- Что кодим: список элементов с фильтром состояния и fallback "пусто".
- Ключевые API: `@if`, `@for`, `@switch`.
- Проверка: корректный рендер для пустого и непустого списка.
- Итог: страница с логикой условного UI.

### 5) Track в `@for` (`/lab/for-track`)
- Цель: понять, зачем нужен `track`.
- Что кодим: список карточек с добавлением и удалением.
- Ключевые API: `@for (...; track item.id)`.
- Проверка: список стабильно обновляется при изменениях.
- Итог: страница с логикой эффективного рендера коллекций.

### 6) Входы и выходы компонента (`/lab/input-output`)
- Цель: освоить современный стиль взаимодействия компонентов.
- Что кодим: родитель и дочерняя карточка, событие удаления.
- Ключевые API: `input()`, `output()`.
- Проверка: дочерний компонент сообщает событие родителю.
- Итог: страница с логикой parent-child коммуникации.

### 7) `model()` и двухсторонняя модель (`/lab/model-two-way`)
- Цель: понять современную двустороннюю связь через модель.
- Что кодим: кастомный мини-контрол "priority selector".
- Ключевые API: `model()`, обновление значения внутри компонента.
- Проверка: значение синхронизируется между parent и child.
- Итог: страница с логикой двухсторонней модели.

### 8) Content projection (`/lab/content-projection`)
- Цель: научиться переиспользовать контейнерные компоненты.
- Что кодим: `card-shell` с `ng-content`.
- Ключевые API: projection, composition.
- Проверка: один shell принимает разный контент без переписывания логики.
- Итог: страница с логикой гибкой композиции.

### 9) Pipes и custom pipe (`/lab/pipes`)
- Цель: работать с форматированием данных.
- Что кодим: список заказов с датой, суммой и статусом.
- Ключевые API: built-in pipes, собственный pipe.
- Проверка: формат данных единообразный во всех карточках.
- Итог: страница с логикой преобразования отображаемых данных.

## Блок B. Роутинг и навигация (страницы 10-17)

### 10) Базовый роутинг (`/lab/router-basics`)
- Цель: настроить несколько страниц в SPA.
- Что кодим: `home`, `about`, `help`.
- Ключевые API: `Routes`, `provideRouter`, `RouterOutlet`, `routerLink`.
- Проверка: переходы работают без перезагрузки страницы.
- Итог: страница с логикой клиентской навигации.

### 11) Параметры пути (`/lab/product/:id`)
- Цель: читать параметры маршрута.
- Что кодим: карточка товара по `id`.
- Ключевые API: `ActivatedRoute`, `paramMap`.
- Проверка: при смене `id` меняются данные страницы.
- Итог: страница с логикой динамического контента по URL.

### 12) Query params (`/lab/search`)
- Цель: передавать и читать параметры фильтров.
- Что кодим: фильтр каталога по `?q=&status=`.
- Ключевые API: `queryParamMap`, `router.navigate`.
- Проверка: URL отражает фильтр и восстанавливает состояние при reload.
- Итог: страница с логикой фильтрации через URL.

### 13) Redirect и wildcard (`/lab/not-found`)
- Цель: обрабатывать некорректные адреса.
- Что кодим: redirect с устаревшего пути и `**` страницу.
- Ключевые API: redirect routes, wildcard.
- Проверка: неверные URL всегда ведут к предсказуемому экрану.
- Итог: страница с логикой fail-safe навигации.

### 14) Nested routes (`/lab/account/*`)
- Цель: освоить дочерние маршруты.
- Что кодим: `account/profile`, `account/security`, `account/billing`.
- Ключевые API: `children`, вложенный `router-outlet`.
- Проверка: layout "account" общий, контент внутри меняется.
- Итог: страница с логикой вложенной навигации.

### 15) Route-level providers (`/lab/admin/*`)
- Цель: понять scoped DI на уровне роутов.
- Что кодим: `admin` раздел с собственным сервисом конфигурации.
- Ключевые API: `providers` внутри route.
- Проверка: сервис доступен только внутри `admin` дерева.
- Итог: страница с логикой локализованных зависимостей.

### 16) Guards (`/lab/private`)
- Цель: ограничить доступ к маршрутам.
- Что кодим: `login` и `private` с `canActivate`.
- Ключевые API: `CanActivateFn`, возврат `UrlTree`.
- Проверка: неавторизованный пользователь уходит на `/login`.
- Итог: страница с логикой контроля доступа.

### 17) Resolver (`/lab/order/:id`)
- Цель: загружать данные до показа страницы.
- Что кодим: заказ с предварительной загрузкой через `ResolveFn`.
- Ключевые API: resolver, `data` в `ActivatedRoute`.
- Проверка: страница получает готовые данные до рендера основного UI.
- Итог: страница с логикой prefetch перед навигацией.

## Блок C. DI, сервисы и HTTP (страницы 18-26)

### 18) Сервис и `providedIn: 'root'` (`/lab/service-root`)
- Цель: вынести бизнес-логику из компонента.
- Что кодим: `todo.service` с CRUD в памяти.
- Ключевые API: `@Injectable`, внедрение через `inject()`.
- Проверка: несколько компонентов читают единый источник данных.
- Итог: страница с логикой через сервисный слой.

### 19) InjectionToken конфиг (`/lab/config-token`)
- Цель: типобезопасная конфигурация приложения.
- Что кодим: токен `APP_CONFIG` с API URL и feature flags.
- Ключевые API: `InjectionToken`, `useValue`.
- Проверка: сервис получает конфиг через DI, без хардкода.
- Итог: страница с логикой конфигурирования через DI.

### 20) HTTP setup (`/lab/http-setup`)
- Цель: корректно подключить HTTP API.
- Что кодим: стартовый запрос списка пользователей.
- Ключевые API: `provideHttpClient`.
- Проверка: сервис делает GET и возвращает typed response.
- Итог: страница с логикой первого HTTP слоя.

### 21) HTTP состояния UI (`/lab/http-state`)
- Цель: правильно показать loading, data, error.
- Что кодим: таблица данных с тремя состояниями.
- Ключевые API: `HttpClient`, `Observable`, `catchError`.
- Проверка: при 500/404 UI не "ломается", а показывает controlled error.
- Итог: страница с логикой устойчивого UI к API-ошибкам.

### 22) Functional interceptor (`/lab/http-interceptor`)
- Цель: централизованно изменять запросы и ответы.
- Что кодим: добавление `Authorization` и логирование latency.
- Ключевые API: functional interceptor, `withInterceptors`.
- Проверка: заголовок и лог появляются без копипаста по сервисам.
- Итог: страница с логикой middleware-уровня HTTP.

### 23) Retry и timeout (`/lab/http-resilience`)
- Цель: базовая отказоустойчивость API-вызовов.
- Что кодим: повтор запроса и таймаут для медленного API.
- Ключевые API: RxJS operators (`retry`, `timeout`, `finalize`).
- Проверка: зависший запрос корректно завершается с понятным сообщением.
- Итог: страница с логикой устойчивости к сетевым сбоям.

### 24) Параллельные запросы (`/lab/http-combine`)
- Цель: собирать страницу из нескольких endpoint.
- Что кодим: dashboard с `profile + stats + notifications`.
- Ключевые API: `forkJoin`, `map`.
- Проверка: dashboard показывает агрегированную view-model.
- Итог: страница с логикой композиции API-ответов.

### 25) Отмена запросов (`/lab/http-cancel`)
- Цель: избежать гонок при быстром вводе.
- Что кодим: live-search по каталогу.
- Ключевые API: `valueChanges`, `debounceTime`, `switchMap`.
- Проверка: старые запросы не перезаписывают новые результаты.
- Итог: страница с логикой безопасного поиска.

### 26) Тестирование HTTP (`/lab/http-test`)
- Цель: писать unit-тесты на сервисы API.
- Что кодим: тесты сервиса данных с мок-бэкендом.
- Ключевые API: `provideHttpClientTesting`, `HttpTestingController`.
- Проверка: тесты валидируют URL, метод и тело запроса.
- Итог: страница с логикой тестируемого HTTP слоя.

## Блок D. Формы (страницы 27-33)

### 27) Template-driven form (`/lab/form-template`)
- Цель: понять базовую форму через template-driven подход.
- Что кодим: форма обратной связи.
- Ключевые API: `FormsModule`, `ngModel`.
- Проверка: валидация required/email работает в шаблоне.
- Итог: страница с логикой простой формы.

### 28) Reactive form basics (`/lab/form-reactive`)
- Цель: перейти на model-driven формы.
- Что кодим: форма регистрации с `FormGroup`.
- Ключевые API: `ReactiveFormsModule`, `FormControl`, `Validators`.
- Проверка: submit активен только при валидной форме.
- Итог: страница с логикой реактивной формы.

### 29) Cross-field validation (`/lab/form-cross-validator`)
- Цель: сравнивать поля формы между собой.
- Что кодим: `password` и `confirmPassword`.
- Ключевые API: custom validator на уровне `FormGroup`.
- Проверка: ошибка показывается только при несовпадении.
- Итог: страница с логикой межполевой валидации.

### 30) Dynamic form array (`/lab/form-array`)
- Цель: динамически добавлять/удалять группы полей.
- Что кодим: список контактов внутри профиля.
- Ключевые API: `FormArray`, `FormGroup`.
- Проверка: добавление и удаление строк сохраняет валидность формы.
- Итог: страница с логикой динамической формы.

### 31) Custom ControlValueAccessor (`/lab/form-cva`)
- Цель: научиться интегрировать собственный UI-контрол в формы.
- Что кодим: рейтинг-звезды как form control.
- Ключевые API: `ControlValueAccessor`.
- Проверка: кастомный контрол участвует в валидации и submit.
- Итог: страница с логикой расширяемых форм.

### 32) Серверные ошибки в форме (`/lab/form-server-errors`)
- Цель: обработка backend validation на UI.
- Что кодим: маппинг ошибок API на поля формы.
- Ключевые API: `setErrors`, error mapping strategy.
- Проверка: ошибка "email already exists" показывается у нужного поля.
- Итог: страница с логикой end-to-end валидации.

### 33) Signal Forms (optional, experimental) (`/lab/form-signals`)
- Цель: познакомиться с новым подходом форм на сигналах.
- Что кодим: мини-форму с signals forms API.
- Ключевые API: пакет `@angular/forms/signals`.
- Проверка: состояние формы реактивно и типобезопасно.
- Ограничение: использовать как исследовательский модуль, не как production-базу.
- Итог: страница с логикой экспериментального form-подхода.

## Блок E. Реактивность, производительность и тесты (страницы 34-40)

### 34) Signals basics (`/lab/signals-basics`)
- Цель: освоить `signal`, `computed`, `effect`.
- Что кодим: калькулятор с derived-state.
- Ключевые API: writable signal, computed signal, effect.
- Проверка: derived значения обновляются без ручного sync-кода.
- Итог: страница с логикой signals.

### 35) OnPush + signals (`/lab/onpush-signals`)
- Цель: понять современную стратегию производительности.
- Что кодим: карточки с фильтрацией и сортировкой.
- Ключевые API: `ChangeDetectionStrategy.OnPush`, signals.
- Проверка: UI обновляется детерминированно, без лишних перерисовок.
- Итог: страница с логикой оптимизированного рендера.

### 36) Deferred loading в шаблоне (`/lab/defer`)
- Цель: отложенная загрузка тяжелых блоков.
- Что кодим: тяжелый график за `@defer` с `@placeholder`.
- Ключевые API: `@defer`, `@placeholder`, triggers (`viewport`, `interaction`).
- Проверка: тяжелый блок не грузится до триггера.
- Итог: страница с логикой шаблонного lazy-loading.

### 37) Route lazy loading (`/lab/lazy-feature`)
- Цель: разрезать приложение на lazy-фичи.
- Что кодим: feature `reports` с отдельным роутом и компонентом.
- Ключевые API: `loadComponent` или `loadChildren`.
- Проверка: chunk фичи грузится при первом переходе.
- Итог: страница с логикой ленивой загрузки по маршрутам.

### 38) Unit testing компонентов (`/lab/test-component`)
- Цель: тестировать компонент и пользовательские действия.
- Что кодим: тесты для страницы со счетчиком и формой.
- Ключевые API: `TestBed`, компонентные фикстуры, `ng test`.
- Проверка: тесты проверяют DOM и состояние после событий.
- Итог: страница с логикой проверяемого UI.

### 39) Unit testing роутинга (`/lab/test-routing`)
- Цель: тестировать навигацию и guards.
- Что кодим: тест guard и редиректа на login.
- Ключевые API: `provideRouter`, тестирование навигации.
- Проверка: приватный маршрут недоступен без авторизации.
- Итог: страница с логикой тестируемой навигации.

### 40) Финальная "операционная" страница (`/lab/ops-dashboard`)
- Цель: собрать одну насыщенную страницу как мини-экзамен.
- Что кодим: данные API + фильтр + форма + error banner + retry.
- Ключевые API: routing, service, http, rxjs, forms, signals.
- Проверка: сценарии success/fail/cancel покрыты вручную и тестами.
- Итог: страница с комплексной логикой реального рабочего экрана.
